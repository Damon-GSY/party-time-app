# Party Time App — 用户体验改进方案

## 一、设计理念

**一句话原则：零学习成本的偏好表达。**

现有交互是"点击选中"——这不能改。偏好表达必须在同一个点击动作里自然延伸，而不是加按钮、加菜单、加模式切换。

核心判断：
- 3 级足够。4 级（必须/有空/勉强/不行）的认知负荷不值得那点额外信息。"必须"是隐含的——你选它为"有空"就是在说可以。
- "不行"就是"不选"——不需要一个单独的状态来表示"我没选这个格子"。
- 最终：**有空（绿）/ 勉强（黄）/ 未选（空）**，3 个可见状态，0 个新操作。

---

## 二、偏好等级设计

### 2.1 三级评分体系

| 状态 | 用户操作 | 视觉 | 分值 | 含义 |
|------|---------|------|------|------|
| 未选 | 默认/第三次点击 | 暗色空格 | 0 | 不方便 |
| 有空 | 第一次点击 | 绿色填充 + ✓ | 2 | 方便 |
| 勉强 | 第二次点击 | 黄色填充 + ~ | 1 | 能来但不方便 |

### 2.2 为什么不是 4 级

"必须有空"这个选项在聚会场景下几乎无意义——如果你标记了某个时段为"有空"，那它就是可以的。额外加一级"必须"只会：

- 增加点击次数（多数用户永远用不到）
- 制造焦虑（"我是不是该标记一些为必须？"）
- 让热力图多一个颜色层次，视觉噪音增大

3 级是信息增益和认知成本的最优平衡点。

## 三、投票交互方案

### 3.1 核心交互：循环点击

```
点击循环：未选 → 有空(绿) → 勉强(黄) → 未选
```

**为什么是循环点击而不是长按/滑动/菜单：**

1. 用户已经会点击——零学习成本
2. 颜色变化本身就是说明——绿色是"好"，黄色是"凑合"
3. 一次点击一个状态，不会误操作
4. 单手可操作，符合地铁/走路场景

### 3.2 视觉状态设计

**未选状态（默认）：**
- 背景：`var(--bg-card)` 即 `#0f3460`（与当前一致）
- 无图标

**有空状态：**
- 背景：`#4ade80`（绿，对应 `--success-color`）15% 透明度填充
- 边框：`#4ade80` 1px 实线
- 图标：白色 ✓（对勾）
- 文字：时段数字变为白色
- 点击动画：当前 `slotSelect` 动画（缩放 1.0→1.15→1.0）

**勉强状态：**
- 背景：`#fbbf24`（黄，对应 `--warning-color`）15% 透明度填充
- 边框：`#fbbf24` 1px 虚线（区别于实线——暗示"不确定"）
- 图标：白色 ~（波浪号）
- 点击动画：同上

### 3.3 状态切换微反馈

每次状态切换时，在格子中心短暂浮现状态文字标签（0.8 秒淡出）：

```
点击进入"有空"：格子中心浮现 "有空" 文字，0.8s 后淡出
点击进入"勉强"：格子中心浮现 "勉强" 文字，0.8s 后淡出
点击回到"未选"：无文字（回到空状态不言自明）
```

这个微反馈解决"用户不知道有第二个状态"的问题——第一次点击时看到"有空"标签，自然会好奇再点会怎样。

### 3.4 快捷操作调整

| 按钮 | 行为 |
|------|------|
| 全天有空 | 将当天所有格子标记为"有空"（绿） |
| 清空当天 | 清除当天所有格子（回到未选） |

没有"全天勉强"——没有人一天都勉强。快捷操作只覆盖 90% 用例。

### 3.5 进度指示调整

当前进度条显示"已选 X 个时段"。调整为：

```
进度显示：已选 8 个时段 · 其中 2 个勉强
进度条：绿色段(有空数量) + 黄色段(勉强数量) 在同一进度条上
```

### 3.6 提交确认

提交前预览（新增）：

```
┌─────────────────────────────┐
│  确认提交                     │
│                              │
│  🟢 有空：6 个时段           │
│  🟡 勉强：2 个时段           │
│                              │
│  周六 14:00-16:00  🟢 有空   │
│  周六 16:00-18:00  🟡 勉强   │
│  周六 18:00-20:00  🟢 有空   │
│  ...                         │
│                              │
│  [修改]        [确认提交]     │
└─────────────────────────────┘
```

## 四、结果展示方案

### 4.1 偏好加权热力图

**评分算法：**

```
每个时段的偏好得分 = (有空人数 × 2 + 勉强人数 × 1)
最高可能得分 = 总参与人数 × 2
归一化得分 = 偏好得分 / 最高可能得分
```

**颜色映射（5 级渐变）：**

| 归一化得分 | 颜色 | 说明 |
|-----------|------|------|
| 0% | `#1a1a2e`（背景色） | 无人可用 |
| 1-25% | `#1e3a5f`（冷蓝） | 少数人，多数勉强 |
| 26-50% | `#2d6a4f`（青绿） | 半数人方便 |
| 51-75% | `#fbbf24`（金黄） | 多数人方便 |
| 76-100% | `#e94560`（主题红） | 几乎所有人都方便 |

用 5 个颜色而不是 3 个，因为结果页面不需要用户做选择——信息密度越高越好。

**格子内容：**

```
┌──────────┐
│  14:00    │
│   █ 87%   │   ← 偏好得分百分比
│  5✓ 2~    │   ← 有空5人，勉强2人（小字）
└──────────┘
```

点击格子展开详情（复用现有 modal）：

### 4.3 共识度指标

在结果页面顶部统计卡片区域，新增"共识度"指标：

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 参与人数  │  │ 最佳时段  │  │ 共识度   │
│    8      │  │   3个     │  │   87%   │
│   人      │  │   候选    │  │  高共识  │
└──────────┘  └──────────┘  └──────────┘
```

**共识度计算：**

```
共识度 = 所有人的"满意度"的平均值
其中，对于某个推荐时段 T：
  用户的满意度 = {
    1.0  如果 T 对该用户是"有空"
    0.5  如果 T 对该用户是"勉强"
    0.0  如果 T 对该用户是"未选"
  }
```

共识度颜色：
- `> 80%`：绿色 `#4ade80` + "高共识"
- `50-80%`：黄色 `#fbbf24` + "中等共识"
- `< 50%`：红色 `#f87171` + "低共识"

### 4.4 帕累托最优标记

**帕累托最优的定义（用户可理解的表述）：**
"没有另一个时段能让更多人方便，同时不让任何人更不方便。"

**可视化方式：**

在热力图和推荐卡片中，帕累托最优时段获得特殊标记：

```
┌──────────┐
│ ★ 14:00  │   ← 金色星标
│   94%    │
│  7✓ 1~   │
└──────────┘
```

标记样式：格子左上角金色小星标 `★`，颜色 `#fbbf24`。

推荐卡片中的标签：圆角药丸标签，金色边框，文字"帕累托最优"。

**不做散点图。** 手机屏幕上散点图是灾难。用标记+文字说明即可。

```
└───────────────────────────────────────┘
```

### 5.3 不做对比展示

**为什么不做"偏好前 vs 偏好后"的对比：**

1. 大多数用户不在乎投票算法，他们只要一个靠谱的推荐
2. 对比展示需要两个推荐结果，计算复杂且 UI 混乱
3. 如果新推荐确实更好，用户自然会感受到——不需要额外的教育

如果要做，仅在"关于"或"设置"页面放一个静态说明，不要干扰主流程。

## 六、数据兼容方案

### 6.1 数据结构扩展

**现有结构：**
```javascript
{
  eventId: "event_id",
  nickname: "小明",
  slots: ["2026-04-04_3", "2026-04-04_4"],
  // slots 数组中的都是"有空"
}
```

**扩展后结构：**
```javascript
{
  eventId: "event_id",
  nickname: "小明",
  slots: ["2026-04-04_3", "2026-04-04_4", "2026-04-05_2"],
  // 保留，兼容旧代码。包含所有"有空"的时段

  preferences: {
    "2026-04-04_3": 2,   // 有空，分值 2
    "2026-04-04_4": 2,   // 有空，分值 2
    "2026-04-05_2": 1    // 勉强，分值 1
    // 未列出的 = 0（不方便）
  }
}
```

### 6.2 兼容规则

| 场景 | 处理方式 |
|------|---------|
| 旧数据（无 preferences 字段） | slots 中所有时段视为"有空"（分值 2） |
| 新数据（有 preferences 字段） | 使用 preferences 计算分数 |
| 混合数据（一次投票中有人用旧版有人用新版） | 分别处理，旧版 slots=available，新版用 preferences |
| 旧版客户端读取新数据 | 忽略 preferences 字段，用 slots 展示（功能降级但不断裂） |

**核心原则：旧版客户端不会崩溃，旧数据不会丢失。**

### 6.3 迁移策略

不需要数据迁移。在读取时按有无 `preferences` 字段判断即可：

```javascript
function getPreferenceLevel(response, slotId) {
  if (response.preferences && response.preferences[slotId] !== undefined) {
    return response.preferences[slotId]; // 新数据：使用偏好分值
  }
  // 旧数据：slots 中存在即为"有空"，否则为"不行"
  return response.slots.includes(slotId) ? 2 : 0;
}
```

### 7.2 图标规范

| 用途 | 图标 | 来源 |

## 八、交互动效规范

### 8.1 状态切换动画

```
未选 → 有空：
  1. 格子缩放 1.0 → 1.1 → 1.0（复用 slotSelect 动画，200ms）
  2. 绿色背景淡入（150ms）
  3. ✓ 图标缩放弹入（100ms 延迟，200ms 动画）
  4. "有空"文字浮现（0ms 延迟，300ms 显示，500ms 淡出）

有空 → 勉强：
  1. 轻微震动反馈（15ms 振动）
  2. 颜色从绿过渡到黄（200ms）
  3. 边框从实线变为虚线（立即切换）
  4. ✓ 图标淡出，~ 图标淡入（200ms 交叉）
  5. "勉强"文字浮现（同上）

勉强 → 未选：
  1. 格子缩放 1.0 → 0.95 → 1.0（轻微回弹，150ms）
  2. 黄色背景淡出（150ms）
  3. 所有元素淡出
```

### 8.2 热力图动效

```
进入结果页时：
  1. 热力图格子依次从左到右、从上到下淡入
  2. 每个格子延迟 30ms，总动画不超过 600ms
  3. 颜色从暗色过渡到目标色（300ms）

推荐卡片：
  1. 卡片从底部滑入（400ms，cubic-bezier(0.16, 1, 0.3, 1)）
  2. 满意度条从左到右展开（500ms，延迟 200ms）
  3. 帕累托标记最后出现（金色闪烁，300ms）
```

### 8.3 触觉反馈

| 操作 | 反馈 |
|------|------|
| 切换到"有空" | 轻振动（15ms） |
| 切换到"勉强" | 中等振动（25ms）——提醒"这不是普通选择" |
| 切换回"未选" | 无振动——取消操作不需要确认感 |
| 提交投票 | 重振动（50ms）+ 成功音 |

### 9.1 投票流程

```

    │

    │     └── 显示引导 overlay → 点击"知道了"→ 关闭
    │
    ├── 选择昵称（与当前一致）
    │
    ├── 浏览时间格子
    │     │
    │     ├── 点击空格子 → 变绿（有空）
    │     ├── 点击绿格子 → 变黄（勉强）
    │     ├── 点击黄格子 → 变空（取消）
    │     │
    │     ├── 点击"全天有空" → 当天全部变绿
    │     └── 点击"清空当天" → 当天全部变空
    │
    ├── 查看进度（N 个有空 + M 个勉强）
    │
    └── 点击提交
          └── 显示确认预览 → 确认 → 提交成功（confetti）
```
